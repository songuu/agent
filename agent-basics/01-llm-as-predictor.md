# B1 · LLM 是预测器，不是数据库

> 目标：先纠正最容易错的心智模型。LLM 不是事实数据库，也不是会自动联网的知识库；它是在给定上下文里预测下一个 token 的模型。

## 核心判断

| 误解 | 更准确的说法 | 工程后果 |
|------|--------------|----------|
| “模型知道答案” | 模型生成看起来像答案的文本 | 重要事实必须检索、校验、引用 |
| “模型会记住上次对话” | 模型只看本次请求带入的 messages | 历史需要由应用回灌 |
| “模型会执行我的要求” | 模型只输出文本或工具调用请求 | 执行权必须在本地代码 |
| “模型说得自信就是真的” | 置信语气不是事实保证 | 需要 eval、来源和拒答策略 |

## 为什么会编造

LLM 的训练目标是让输出在语境里高概率成立，不是保证每个断言都可追溯。它可以把真实模式、近似概念、错误记忆和上下文暗示混成一段很流畅的话。

工程上不要问“怎么让模型永远不编造”，而是问：

- 这类问题是否需要外部事实源。
- 事实源能否被检索、引用、版本化。
- 没有证据时应该拒答、追问还是降级。
- 输出是否进入高风险动作，比如发邮件、写数据库、执行命令。

## 能力边界

| 能力 | 模型擅长 | 模型不擅长 | 应用层补法 |
|------|----------|------------|------------|
| 语言 | 改写、总结、分类、解释 | 保证每个事实最新 | 接检索、引用和日期 |
| 推理 | 多步拆解、假设比较 | 长链条精确计算 | 工具计算、单元测试 |
| 知识 | 常识、公开模式、训练期知识 | 私有数据、实时状态 | RAG、API、数据库 |
| 计划 | 给出候选步骤 | 判断真实环境是否完成 | 执行器和验收信号 |

## 常见误区

1. 把 prompt 写得更强硬，试图替代事实校验。
2. 让模型“自己检查引用”，但不给它真实来源。
3. 让模型做金额、日期、权限等精确判断，却不调用工具。
4. 将旧训练知识当成当前真实状态。

## 自检练习

拿一个问题：“我们产品今天线上还有多少活跃用户？”

回答下面四个问题：

- 模型单独能不能回答？为什么？
- 需要哪些真实数据源？
- 如果数据源不可用，应该拒答、推断还是追问？
- 输出里哪些内容必须带来源或查询时间？

## 记住一句话

LLM 负责生成候选答案；应用负责提供事实、约束、执行和验证。
