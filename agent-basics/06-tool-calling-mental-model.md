# B6 · Tool Calling 心智模型

> 目标：理解模型不会真的执行工具。它只提出结构化请求；本地代码决定是否执行、怎么校验、如何回传结果。

## 一次工具调用往返

```text
user request
  -> model: 我需要调用 search({"query":"..."})
  -> app: 校验参数和权限
  -> app: 执行 search
  -> app: 回传 tool result
  -> model: 基于结果继续或给最终答案
```

## 职责边界

| 环节 | 模型负责 | 应用负责 |
|------|----------|----------|
| 选择工具 | 提出候选工具调用 | 判断工具是否允许 |
| 生成参数 | 按 schema 填参数 | 校验类型、范围、权限 |
| 执行工具 | 不执行 | 调 API、查库、写文件 |
| 处理错误 | 读取错误 observation | 捕获异常、清洗错误、回传 |
| 终止循环 | 给出最终答案 | maxSteps、预算、人工确认 |

## Tool Schema 是 ABI

Schema 不只是“给模型看的说明”，也是代码执行边界：

- 字段名要有业务语义。
- 描述要说明何时使用。
- enum 优先于自由文本。
- 数值要有范围。
- 高风险参数要二次确认。

## 错误回传

不要把工具错误藏起来。把错误转成模型能理解的 observation：

```text
Tool error: customerId must be a non-empty string. Ask user for a valid customer id.
```

这样下一轮模型才可能追问或修正参数。

## 高风险工具

以下工具需要确认门：

- 发邮件、发消息、创建工单。
- 写数据库、删除文件、执行命令。
- 花钱、下单、改权限。
- 访问隐私数据或跨租户数据。

确认门应该在应用层实现，不要只靠 prompt 请求模型“谨慎”。

## 自检练习

设计一个 `sendEmail` 工具：

- 参数 schema 怎么写？
- 哪些字段需要校验？
- 哪些场景必须人工确认？
- 发送失败怎么回传给模型？

## 记住一句话

模型可以请求工具；执行权永远属于你的代码。
