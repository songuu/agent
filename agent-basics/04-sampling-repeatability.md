# B4 · 采样参数与可重复性

> 目标：理解 temperature、top_p、seed 这些参数不是“玄学旋钮”，而是影响稳定性、创造性和评估可复现性的工程变量。

## 采样在控制什么

模型每一步会给候选 token 一个概率分布。采样参数控制从这个分布里怎么选。

| 参数 | 作用 | 常见用法 |
|------|------|----------|
| temperature | 调整分布尖锐程度 | 低温用于分类/抽取，高温用于创意 |
| top_p | 只在累计概率前 p 的候选里采样 | 控制候选范围 |
| seed | 尽量复现随机过程 | 用于调试和 eval 稳定 |
| max_tokens | 限制最大输出长度 | 防止长输出失控 |
| stop | 遇到指定片段停止 | 简单协议或模板输出 |

## 参数选择

| 任务 | 推荐倾向 | 理由 |
|------|----------|------|
| JSON 抽取 | 低 temperature | 输出结构要稳定 |
| 分类打标 | 低 temperature | label 不该漂移 |
| 代码修改 | 低到中 | 需要稳定且能补全细节 |
| brainstorming | 中到高 | 需要多样候选 |
| 产品文案 | 中 | 兼顾稳定和表达 |
| LLM-as-judge | 低 | 评估要可重复 |

## 可重复性的边界

即使 temperature 设 0，也不代表跨模型版本、跨供应商、跨时间完全一致。原因包括：

- 模型权重或 serving 策略更新。
- 工具返回内容变化。
- 检索结果顺序变化。
- 浮点和并行推理细节。
- prompt 中隐含上下文变化。

所以 eval 不能只依赖“一次输出完全相同”，更适合测结构、关键字段、引用和行为边界。

## 工程建议

1. 生产抽取、路由、打分默认低 temperature。
2. 创意任务可以生成多候选，再用 eval 或规则筛选。
3. eval 记录模型、参数、prompt 版本和数据集版本。
4. 对关键任务测“是否满足契约”，不要测整段文本逐字相同。
