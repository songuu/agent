# B5 · 指令、约束与输出契约

> 目标：把 prompt 从“提示词文案”升级成接口契约。真正可维护的 prompt 需要角色、任务、边界、格式、失败策略和示例。

## Prompt 的组成

| 部分 | 作用 | 示例 |
|------|------|------|
| 身份 | 限定回答视角 | 你是严谨的 TypeScript 课程助教 |
| 任务 | 描述要完成什么 | 将用户需求拆成实现任务 |
| 约束 | 规定不能做什么 | 不要编造 API，不要跳过验证 |
| 格式 | 给下游可解析输出 | 返回 JSON，字段为 title/tasks/risks |
| 示例 | 让模型对齐风格 | 输入 A 输出 B |
| 失败策略 | 不满足条件时怎么办 | 信息不足时列出需要补充的问题 |

## 输出契约

自然语言输出适合教学解释；生产链路更需要契约。

| 输出形态 | 适用 | 风险 |
|----------|------|------|
| 纯文本 | 解释、总结、写作 | 下游难解析 |
| Markdown | 报告、课程内容 | 表格和列表可能漂移 |
| JSON | 抽取、路由、配置 | 需要验证和修复 |
| Tool call | 需要真实执行 | 参数和权限必须校验 |

## 约束写法

弱约束：

```text
请尽量简洁。
```

强约束：

```text
输出必须少于 120 个中文字。若需要更多信息，返回：
{"status":"needs_input","questions":["..."]}
```

强约束有可验证条件和失败分支。

## 常见契约字段

| 字段 | 用途 |
|------|------|
| `status` | success / needs_input / blocked |
| `confidence` | high / medium / low |
| `evidence` | 引用来源或工具结果 |
| `assumptions` | 明确推断 |
| `risks` | 未关闭风险 |
| `next_steps` | 可执行后续动作 |

## 自检练习

把“帮我分析这封邮件重要不重要”改成契约：

- 输出字段有哪些？
- 缺少邮件正文时怎么返回？
- 需要区分事实、推断、建议吗？
- 下游代码怎么判断是否需要提醒用户？

## 记住一句话

Prompt 不是愿望清单；它是模型和应用之间的行为协议。
