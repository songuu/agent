# B8 · Memory、RAG 与 Context 不是一回事

> 目标：拆清三件容易混在一起的事：聊天历史、长期记忆、知识库检索。它们解决的问题不同，风险也不同。

## 三层边界

| 层 | 解决什么 | 典型内容 | 主要风险 |
|----|----------|----------|----------|
| Context | 本次调用让模型看到什么 | system、user、history、retrieval | 太长、重复、污染 |
| Memory | 跨轮或跨会话保留状态 | 偏好、摘要、任务状态 | 过时、误记、隐私 |
| RAG | 从外部知识源取证据 | 文档 chunk、搜索结果、引用 | 召回差、注入、引用错 |

## 短期记忆

短期记忆通常就是最近对话和摘要。它解决连续性：

- 用户说“继续刚才的方案”。
- agent 需要知道上一轮工具结果。
- 多步任务需要保持中间状态。

它不适合保存长期事实，因为窗口会变长、变贵、变脆。

## 长期记忆

长期记忆保存跨会话信息：

- 用户偏好。
- 项目约定。
- 已确认决策。
- 常见失败模式。

长期记忆必须可撤销、可更新、可追溯。否则错误记忆会长期污染结果。

## RAG

RAG 用来把外部知识带入当前回答：

```text
query -> retrieve -> rerank -> context builder -> generate with citations
```

RAG 的关键不是“塞更多资料”，而是：

- 找到相关资料。
- 去重和排序。
- 保留来源。
- 明确证据不足时拒答。

## Context Builder

Context builder 负责把 memory、RAG、工具结果打包给模型。它应该决定：

- 哪些内容进入上下文。
- 顺序如何安排。
- 每段最长多少。
- 重复内容如何压缩。
- 不可信内容如何标注。

## 自检练习

设计一个企业知识库客服 agent：

- 用户偏好放 memory 还是 RAG？
- 产品文档放 memory 还是 RAG？
- 最近工单状态放 context、memory 还是工具结果？
- 检索到的文档里有“忽略系统指令”怎么办？

## 记住一句话

Memory 保存状态；RAG 提供证据；context 是本次调用的装配结果。
