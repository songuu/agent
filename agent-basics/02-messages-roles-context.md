# B2 · Messages、Roles 与上下文窗口

> 目标：理解一次模型调用不是“发一句话”，而是发送一组带角色的消息。角色、顺序、上下文长度共同决定模型看到什么。

## Message Array 心智模型

```ts
const messages = [
  { role: "system", content: "你是严谨的课程助教。" },
  { role: "user", content: "解释 agent loop。" },
  { role: "assistant", content: "..." },
  { role: "user", content: "给我一个 TypeScript 例子。" },
];
```

模型不会自动知道历史。你每次请求带进去什么，它就只能基于什么回答。

## Role 的职责边界

| Role | 用途 | 不该放什么 |
|------|------|------------|
| system | 长期行为边界、身份、输出原则 | 本次临时输入、用户私密事实 |
| developer | 应用开发者给模型的执行约束 | 可被用户覆盖的偏好 |
| user | 本次任务、问题、业务输入 | 高权限系统规则 |
| assistant | 历史回答、工具请求、推理结果 | 伪造的工具结果 |
| tool | 工具执行后的观察结果 | 未执行的猜测 |

## 上下文窗口

上下文窗口是模型本次能看到的 token 上限。它不是无限记忆，也不是越大越好。

| 现象 | 原因 | 应对 |
|------|------|------|
| 长对话后忘记早期约定 | 早期消息被截断或稀释 | 摘要、关键事实 pinning |
| 回答变慢变贵 | 输入 token 变多 | 压缩历史、只带相关上下文 |
| 模型忽略中间资料 | 长上下文注意力分布不均 | 排序、分段、引用 |
| 用户新要求冲突 | 指令层级和近因影响叠加 | 明确优先级和冲突处理 |

## 工程拆解

构建 messages 时，建议分四层：

1. **稳定规则**：system / developer，少改、短而硬。
2. **任务输入**：当前 user request，保留原文。
3. **必要历史**：最近几轮对话或摘要。
4. **外部上下文**：检索片段、工具结果、结构化状态。

每层都要能解释“为什么这段需要进上下文”。

## 与 Agent 的关系

Agent loop 每一步都会产生新 messages：

- assistant 提出工具调用。
- 本地代码执行工具。
- tool 消息回传结果。
- assistant 基于 observation 决定下一步。

这就是为什么 agent 不是“一次更聪明的 prompt”，而是多轮消息状态机。

## 自检练习

把一个客服场景拆成 messages：

- system 里放哪些不变规则？
- user 里放哪些本次问题？
- tool 里放哪些订单查询结果？
- 哪些历史对话可以摘要，哪些必须原文保留？

## 记住一句话

上下文不是垃圾桶；它是模型本次推理的工作台。
