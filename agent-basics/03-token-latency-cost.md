# B3 · Token、延迟与成本直觉

> 目标：建立预算意识。Agent 的质量问题常常不是模型不够强，而是上下文太长、步骤太多、工具太贵、输出太散。

## Token 是什么

Token 是模型处理文本的基本单位。中文、英文、符号、代码都会被切成 token。你不需要背精确算法，但必须知道：

- 输入 token 越多，成本越高。
- 输出 token 越多，延迟越高。
- 多轮 agent loop 会把历史和工具结果继续带入后续请求。
- RAG 检索片段会快速占满上下文。

## 成本来源

| 成本项 | 来自哪里 | 常见失控点 |
|--------|----------|------------|
| 输入 token | system、历史、检索、工具结果 | 原样塞全文、重复塞相同资料 |
| 输出 token | 模型回答、计划、解释 | 让模型无限展开、缺少长度约束 |
| 步数 | agent loop 多轮调用 | 没有 maxSteps、工具失败后反复尝试 |
| 工具 | 搜索、数据库、外部 API | 每轮重复查同一份数据 |
| 评估 | LLM-as-judge、回归集 | 每次提交跑全量高成本 eval |

## 延迟拆解

一次 agent 响应通常包含：

1. 请求排队。
2. 模型首 token 时间。
3. 模型输出时间。
4. 工具调用时间。
5. 后续模型轮次。

流式输出只能改善用户感知，不会减少总计算成本。真正降低成本需要减少上下文、步骤和无效工具调用。

## 预算策略

| 场景 | 建议 |
|------|------|
| 长对话 | 最近 N 轮原文 + 摘要 |
| RAG | top-k 小而精，先 rerank 再打包 |
| 工具循环 | maxSteps + 每类工具调用上限 |
| 结构化输出 | schema 限制字段和长度 |
| eval | smoke 集每次跑，完整集定时或发布前跑 |

## 自检练习

给一个 research agent 设预算：

- 最多搜索几次？
- 每次保留几段资料？
- 每段最长多少 token？
- 最终答案最长多少字？
- 什么时候停止并说明证据不足？

## 记住一句话

Agent 的预算不是上线后才算；它必须写进循环、检索和输出契约。
