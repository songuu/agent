# 🚀 创业指南：用 AI Agent 做产品（从 demo 到能赚钱）

> 面向：学完本课程、想拿 Agent 能力做点真东西（副业 / 创业 / 公司内部工具）的初学者。
>
> 这份指南只讲**可操作的判断与动作**，不灌鸡汤。读完你应该能回答四个问题：
> 我该做哪个场景？ 第一个版本砍到多小？ 一次请求要花多少钱？ 上线前还差什么？

> ⚠️ 一句话前提：**Agent 是手段，不是产品。** 用户为「问题被解决」付费，不为「你用了 LLM」付费。

---

## 一、从 demo 到产品：路径与心态

### 大多数 Agent 项目死在哪

不是死在技术，是死在三件事：

1. **demo 太炫、闭环太弱**：能演示，但没人能靠它**完成一件真实的事**。
2. **成本失控**：上线才发现每个活跃用户每月烧掉的 token 比订阅费还高。
3. **不可靠**：10 次里坏 2 次，用户就不再信任它——而信任一旦没了就回不来。

### 正确的演进顺序

```
玩具 demo            最小可用闭环           可收费产品            可规模化
(能跑通一次)   →    (一类人能反复用)   →   (有人愿意付钱)   →   (单位经济为正)
  1 个周末            2-4 周                持续迭代              再谈增长
```

**心态铁律：先做能用的最小闭环，再谈规模。**

- ❌ 错误顺序：先搭多智能体编排 → 接 5 个数据源 → 做后台管理 → 然后找用户。
- ✅ 正确顺序：找一个**愿意现在就用**的人 → 给他一个能跑通**一件事**的窄工具 → 看他是否**第二天还回来用** → 再加第二件事。

> 判断你是否还在「玩具」阶段的硬指标：**有没有一个你不认识的人，主动用了第二次。** 没有，就别加功能，去改闭环。

### 「能用的最小闭环」长什么样

一个闭环 = 用户带着**一个真实输入**进来，拿到一个**他敢用的输出**出去。例子：

| 场景 | 输入 | 输出（闭环终点） |
|------|------|------------------|
| 合同审查助手 | 一份 PDF 合同 | 风险条款清单 + 出处定位（不是"聊聊这份合同"） |
| 竞品调研 | 一个产品名 | 一页结构化对比报告 + 引用链接（本课程毕业项目就是这个雏形） |
| 客服初筛 | 一封工单邮件 | 分类 + 草拟回复 + 是否需转人工 |

注意：闭环的终点都是**可直接使用的结构化结果**，不是"开放式对话"。对话是 demo，结果才是产品。

## 二、机会判断：什么场景适合用 Agent

不是所有问题都该上 Agent。用下面这把尺子先筛。

### ✅ 适合的场景，同时满足这三条

| 维度 | 含义 | 反问自己 |
|------|------|----------|
| **高价值** | 解决一次能省下可观的人力 / 时间 / 金钱 | 用户愿意为单次结果付多少？低于 token 成本就别做 |
| **可容错** | 错了有补救，不是一锤定音 | 输出错了，最坏后果是什么？能不能人工兜底？ |
| **有数据/工作流** | 存在可复用的资料、规则、流程让 Agent 依附 | 这件事人类是怎么做的？有 SOP / 文档 / 历史数据吗？ |

三条缺一条都要打问号；缺两条基本别做。

### ❌ 不适合的场景

- **零容错 / 高后果**：医疗诊断结论、税务/法律的最终裁定、自动下单付款（除非有强人工确认）。
- **需要 100% 精确且可验证**：精确数值计算、账目核对——这类用确定性代码，别用概率模型。
- **没有任何数据或工作流依附**：纯靠模型"凭空创作"且无从校验对错的，护城河和可靠性都立不住。
- **实时性要求极高**：Agent 多步推理 + 工具调用天然有延迟（秒级到十几秒），不适合毫秒级响应。
- **本质是确定性逻辑**：能用 `if/else` + 一条 SQL 解决的，别套 LLM——又慢又贵又不稳。

> 经验法则：**Agent 擅长「处理模糊、非结构化的输入，产出结构化的中间结果，由人做最终决策」。** 越靠近这个定位越稳，越远越危险。

### 一个 60 秒自检清单

开工前逐条打勾，2 个以上空着就重新想：

- [ ] 我能说清**谁**、在**什么时刻**、为了**什么具体结果**用它。
- [ ] 这个结果对用户值钱（能换算成省下的时间或金钱）。
- [ ] 输出错了不会造成不可逆的灾难，且能人工纠正。
- [ ] 我手上有（或能拿到）让它变准的数据/规则/流程。
- [ ] 我自己能在没有 UI 的情况下，先用命令行跑通这个闭环。

## 三、MVP 范围裁剪：砍到最小可验证

初学者最大的陷阱是「想做完整产品」。MVP 的目标不是完整，是**用最小成本验证「有人要」这一个假设**。

### 裁剪四刀

1. **砍场景**：只服务**一类人**的**一个任务**。不是"通用助手"，是"给跨境电商客服自动分类英文工单"。
2. **砍输入形态**：先只接**一种输入**（先支持上传 PDF，不要同时支持 PDF/网页/图片/语音）。
3. **砍交互**：能用**命令行 / 一个表单 + 一个结果页**验证的，就不要做账户体系、后台、多端。
4. **砍自动化程度**：先做 **human-in-the-loop**（Agent 出草稿、人确认），别一上来全自动。又快又安全。

> 判断砍得够不够狠的标准：**这个 MVP 你能在两周内、不写 UI 也能让真实用户试用吗？** 不能，就再砍。

### 用本课程能力栈快速搭 MVP

本课程从零手写、并沉淀在 `src/shared/` 的能力，正好是一个 MVP 的全部零件。对照着搭：

| MVP 需要的能力 | 用课程的哪块 | 对应章节 / 代码 |
|----------------|--------------|------------------|
| 调模型、厂商无关、随时换便宜模型 | `getLLM()` 统一抽象 | 02 章 · `src/shared/llm/` |
| 让模型按你的话术干活 | 提示工程（system/few-shot/CoT） | 03 章 |
| 让它自己决定调哪个工具、循环到完成 | 手写 agent 循环 (ReAct) | 04 章 · `src/shared/agent/loop.ts` |
| 给它接你的业务能力（查库、调 API） | 工具系统（zod schema + 注册表 + 安全执行） | 05/06 章 · `src/shared/agent/tool.ts` |
| 让它「记住」对话上下文且不爆 token | 短期记忆（滑动窗口 + 摘要压缩） | 07 章 |
| 让它「读你的资料」回答 | 从零 RAG（分块/检索/注入/引用） | 08/09 章 · `src/shared/rag/vectorStore.ts` |
| 复杂任务拆解、多角色协作 | 推理范式 / 多智能体编排 | 10/11 章 |
| 输出必须是稳定 JSON（喂给下游系统） | 结构化输出 + zod 校验 + retry-repair | 13 章 |
| 前端有"在思考/在打字"的体验 | 流式输出与 UX | 14 章 |
| 上线前必须有的护栏 | 评估 / 可观测 / 安全 / 部署 | 15-18 章（见第八节 checklist） |

**最快的 MVP 骨架（伪流程）**：

```
用户输入
  → getLLM()                       // 02：一行换厂商/换便宜模型
  → 带 system 提示的 agent 循环      // 03+04：让它按你的话术干活
  → 调你注册的几个工具(查资料/调API)  // 05/06
  → (可选) RAG 注入你的私有知识       // 08/09
  → 用 zod 把输出锁成结构化 JSON     // 13
  → 返回给用户 / 写回业务系统
```

毕业项目 **Deep Research Agent** 就是这套骨架的完整范例——它本身可以直接改成一个「调研助手」类 MVP 的起点。若你的产品核心是“上传资料 → 知识库问答 → 引用溯源”，则把 [RAG 系统实战项目](./rag-system-project.md) 与 [songuu/rag-system](https://github.com/songuu/rag-system) 作为主线,比从 Deep Research Agent 改造更直接。

## 四、成本与单位经济：把每一分 token 钱算清楚

**不会算成本 = 不知道自己在赔钱。** 这是 Agent 创业和普通 SaaS 最大的区别：你的边际成本不为零，每次请求都在花钱。

### 成本公式（与第 16 章代码一致）

课程第 16 章 `lessons/16-observability-and-cost/pricing.ts` 里的 `estimateCost` 就是这条公式：

```
单次调用费用 = (输入 tokens / 1_000_000) × 输入单价
            + (输出 tokens / 1_000_000) × 输出单价

// 单价单位：美元 / 每百万(1M) tokens
```

课程价格表（**示意价，真实价以厂商官网为准**）：

| 模型（示意） | 输入 $/1M | 输出 $/1M | 定位 |
|------|----------:|----------:|------|
| `claude-opus`（旗舰） | 15 | 75 | 最强推理，最贵 |
| `claude-sonnet`（均衡） | 3 | 15 | 主力 |
| `claude-haiku`（轻量） | 1 | 5 | 高频/简单任务 |
| `gpt-4o` | 2.5 | 10 | 主力 |
| `gpt-4o-mini` | 0.15 | 0.6 | 高频/简单任务 |

> 关键观察：**输出 token 通常比输入贵 4-5 倍**；旗舰比轻量贵一个数量级。这两点决定了你所有的优化方向。

### 一个粗略的「每请求成本」算法示例

不要只算「一次 LLM 调用」——**Agent 一次任务会循环好几轮**（每轮都是一次完整计费）。把整条链路加起来才是真成本：

```ts
// 粗算一次 Agent 任务的成本（伪代码，对齐第 16 章 estimateCost）
function estimateTaskCost() {
  const loops = 4;              // agent 循环平均轮数（思考→调工具→再思考...）
  const inPerLoop = 3_000;      // 每轮平均输入 tokens（含 system+历史+RAG 注入）
  const outPerLoop = 500;       // 每轮平均输出 tokens

  // 用 sonnet 价：输入 $3/1M，输出 $15/1M
  const inCost  = (loops * inPerLoop  / 1_000_000) * 3;   // = 0.000036 → $0.036
  const outCost = (loops * outPerLoop / 1_000_000) * 15;  // = 0.00003  → $0.030
  return inCost + outCost;      // ≈ $0.066 / 次任务
}
```

把它算成**单位经济**才有意义：

```
单次任务成本 ≈ $0.066
假设一个付费用户平均每月跑 200 次任务
→ 每用户月度模型成本 ≈ $13.2
→ 你的订阅价必须 > $13.2 + 服务器/人力分摊 + 利润，否则做一单赔一单
```

> ⚠️ **务必先用命令行 + 第 16 章的 `Tracer` 把真实任务跑几十遍，量出真实的「平均轮数 / 平均 tokens」再定价。** 拍脑袋的估算通常乐观 2-5 倍。

### 四个真正有效的降本手段

1. **模型分级（最大杠杆）**：简单步骤（分类、抽取、格式化）用 haiku / 4o-mini，只有难推理才上 sonnet/opus。一个任务里混用不同模型，整体成本能砍掉一半以上。
2. **Prompt 缓存（Prompt Caching）**：把固定不变的 system 提示 / 长文档 / few-shot 标记为可缓存，命中后这部分输入只按零头计费。对「同一份长上下文反复问」的场景立竿见影。
3. **RAG 降上下文**：不要把整本手册塞进 prompt。用第 08/09 章的检索，只取**最相关的几段**注入——既省输入 token，又减少幻觉。这是「输入侧」最大的省钱点。
4. **控制循环轮数 / 压缩历史**：给 agent 循环设**最大轮数**上限（防失控空转）；用第 07 章的滑动窗口 + 摘要压缩长对话历史，避免历史越滚越长、每轮越来越贵。

> 顺序记忆：**先 RAG 砍输入 → 再分级砍贵模型 → 再缓存砍重复 → 最后压历史控轮数。**

## 五、技术选型：自建 vs 框架 vs 平台

没有「最好」，只有「现阶段最划算」。三条路的权衡：

| 维度 | 自建（本课程方式） | 框架（LangGraph / Vercel AI SDK） | 平台 / 低代码（Dify / Coze 等） |
|------|--------------------|-----------------------------------|--------------------------------|
| 上手速度 | 慢（要写循环/工具/记忆） | 中（封装好了循环与状态） | 快（拖拽即用） |
| 控制力 | 最高（每一行你都懂） | 高（可定制，但要懂其抽象） | 低（受平台能力边界限制） |
| 成本可控性 | 最高（自己掐每次调用） | 高 | 中/低（平台抽成、黑盒计费） |
| 调试/可观测 | 最透明 | 需接其 trace 体系 | 依赖平台面板 |
| 厂商锁定风险 | 最低（厂商无关抽象） | 低-中 | 高（迁出成本大） |
| 适合阶段 | 学习原理 / 核心逻辑 / 要极致控本 | 产品化、状态复杂、要快又要可控 | 极早期验证 / 非技术团队 / 内部工具 |

**给初学创业者的务实建议：**

- **验证期**：能用平台/低代码 1 天搭出来验证"有没有人要"，就先用，别纠结技术栈。验证假设最重要。
- **产品期**：核心闭环（agent 循环、工具、计费、护栏）建议**自建或用框架**——因为成本、可靠性、护城河都在这里，你必须能掌控。
- **混合是常态**：用框架管编排（LangGraph 的状态机），但工具层、计费层、护栏层自己写。本课程「先手写懂原理、再上框架」正是为这种混合打底——**你用框架时知道它在替你做什么，出了问题能下沉到原理层去查。**

> 反模式：还没验证需求，就先花两周纠结 LangGraph 还是自建。**选型是产品期的问题，不是验证期的问题。**

## 六、护城河与数据飞轮：为什么「套壳」不够

**「套壳」= 只是把用户输入转发给 OpenAI/Claude，再把回答转发回去。** 它的致命问题：

- 任何人都能在一周内复制你的提示词。
- 模型厂商可能直接出官方功能把你覆盖掉。
- 你没有任何随时间增厚的东西——今天和明天的你一样脆弱。

护城河不在模型（模型是大家共用的），在你**围绕模型构建的、别人短期拿不走的东西**：

### 三类可积累的壁垒

1. **数据（最硬）**：用户在你这里产生的、别处没有的专有数据——历史工单、标注好的对错样本、行业知识库。它直接喂回去让你的 Agent 更准。
2. **工作流嵌入**：你的 Agent 深度嵌进用户的真实流程（对接了他们的 CRM、走了他们的审批链）。嵌得越深，迁移成本越高，越离不开你。
3. **分发**：你能持续、低成本地触达目标用户的渠道（社区、垂直内容、合作生态）。再好的产品没分发也死。

### 数据飞轮怎么转

这是 Agent 产品最值得追求的复利结构：

```
更多用户使用
   ↓
产生更多「真实输入 + 用户对结果的反馈/修正」
   ↓
沉淀成你的专有数据（评估集 / few-shot 样本 / 微调数据 / 知识库）
   ↓
Agent 变得更准、更懂这个细分场景
   ↓
体验更好 → 留住老用户、吸引新用户
   ↓ （回到顶部，越转越快）
```

落地动作（用本课程的能力就能埋下飞轮第一环）：

- 把用户**点过「这条不对/已修正」**的样本，沉淀进第 15 章的**评估集**——既防回归，又是未来微调的燃料。
- 用第 16 章的 **Tracer** 记录每次任务的输入/输出/成本，这些日志本身就是你最早的专有数据资产。
- 用第 09 章的 **RAG 知识库**承接用户/行业的私有资料——别人没有这份库，就复制不了你的效果。

> 一句话：**「套壳」是起点不是终点。第一版可以是壳，但从第一天起就要想清楚：我的数据/工作流/分发壁垒，从哪一环开始攒？**

## 七、常见坑（按踩中频率排序）

| 坑 | 表现 | 怎么防 |
|----|------|--------|
| **幻觉与可靠性** | 模型一本正经地编造事实/出处 | RAG 限定它只能基于检索内容回答（09 章）；输出强制带引用/出处；用 zod 锁结构（13 章）；高后果步骤加人工确认（17 章） |
| **评估缺失** | 改了 prompt 自我感觉变好了，其实悄悄坏了别的 | 从第一天就建评估集 + 通过率门槛（15 章），别等出事才补 |
| **成本失控** | 上线后账单暴涨，单位经济为负 | 上线前用 Tracer 量真实成本（16 章）；模型分级 + RAG 降上下文 + 缓存 + 限循环轮数；按用量设熔断/限额 |
| **合规与隐私** | 把用户敏感数据/PII 直接发给第三方模型 | 明确数据流向并告知用户；敏感字段脱敏后再入 prompt；评估是否需要数据不出境 / 私有部署；查清厂商的数据保留与训练条款 |
| **Prompt 注入** | 用户/外部网页内容里藏指令，劫持你的 Agent | 隔离不可信内容、加输入护栏、限制工具权限（17 章 prompt injection） |
| **过度依赖单一厂商** | 厂商涨价/限流/下线模型，你被卡脖子 | 坚持厂商无关抽象（本课程的 `getLLM()`），核心逻辑不绑死某家；关键路径准备备选模型 |
| **过度工程** | 还没验证需求就搭多智能体 + 全套基建 | 先 MVP 闭环（第三节四刀），需求验证后再加复杂度 |
| **延迟劝退用户** | 多步推理一等十几秒，用户以为卡死 | 流式输出 + 步骤可见（14 章）；能并行的工具调用并行；简单路径走快模型 |

## 八、上线 checklist（结合 15/16/17/18 章）

上线不是"代码跑通了"，是"出了问题你能发现、能兜底、能复盘"。逐项打勾：

### ✅ 评估（第 15 章 · 质量底线）

- [ ] 有一份覆盖核心场景 + 边界 case 的**离线评估集**。
- [ ] 评估能**自动打分**（规则优先，主观维度才上 LLM-as-judge）。
- [ ] 评估接入 **CI**：通过率低于门槛就让流水线变红，**退化的 prompt 进不了主干**。
- [ ] 每修一个线上 bad case，都补一条**回归用例**（永不删除）。

### ✅ 可观测性与成本（第 16 章 · 看得见才管得了）

- [ ] 每次任务都用 **Tracer** 记录：模型 / 输入输出 tokens / 耗时 / 费用。
- [ ] 算清并持续监控**单位经济**（每用户 / 每任务真实成本 vs 收入）。
- [ ] 设了**成本熔断 / 用量限额**，防单用户或失控循环烧爆账单。
- [ ] 已应用降本：**模型分级 + RAG 降上下文 +（可用则）prompt 缓存 + 限循环轮数**。

### ✅ 安全与护栏（第 17 章 · 别让它闯祸）

- [ ] **无硬编码密钥**：API key 全走环境变量，绝不进代码/前端/日志。
- [ ] 有 **prompt injection 防护**：隔离不可信内容、加输入/输出护栏。
- [ ] 工具按**最小权限**授予；危险/不可逆操作（付款、删除、外发）必须**人工确认**。
- [ ] 所有外部输入**校验**后再进 prompt（zod / schema）；敏感数据脱敏。
- [ ] 明确数据流向与隐私边界，符合目标市场的合规要求。

### ✅ 部署（第 18 章 · 变成真正的服务）

- [ ] Agent 包成 **HTTP API / 服务**，不是只能本地 `tsx` 跑的脚本。
- [ ] 长任务有**流式返回（SSE）**，前端能看到进度，不假死。
- [ ] 有**超时、重试、限流**与优雅降级（模型挂了有兜底回复）。
- [ ] 有**结构化日志**与基本告警（错误率 / 延迟 / 成本异常）。
- [ ] 跑过一遍第 18 章的部署清单，确认环境变量、依赖、启动命令在目标环境就绪。

> 全部打勾再上线。每一条都对应着一类"上线后才暴雷、修起来十倍痛"的问题——**checklist 的价值就是把痛苦从生产环境提前到上线前。**

## 九、行动起点（今天就能做的三件事）

1. **写一句话假设**：把你的想法压成一句——"给【某类人】在【某时刻】自动产出【某结果】"。说不清就回第二节自检。
2. **不写 UI，先跑闭环**：用本课程的 `getLLM()` + agent 循环 + 几个工具，在命令行把这件事跑通一次。跑不通，说明想法还没成立。
3. **找一个真实用户试**：把命令行结果发给一个**真有这个需求的人**。他用了第二次，你就有产品了；他没用，你省下了几个月。

> 最后记住：**Agent 创业拼的不是谁的模型聪明（模型大家共用），是谁更早把「可靠的闭环 + 正的单位经济 + 在转的数据飞轮」三件事跑通。** 技术只是入场券，这三件事才是牌桌上的筹码。
